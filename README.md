# 撞球顏色辨識器

一個用 Three.js 做的網頁小工具：點選一顆撞球，畫面右邊會出現一顆可以自由旋轉的 3D 球，
下方會即時告訴你「根據目前鏡頭正對的那一面」，這顆球的顏色組成對應到幾號球——**完全不
靠讀取印在球上的數字**，純粹用顏色跟圖案分析來判斷。

這份 README 的目標不是教你怎麼用（用法很簡單，點一下球就好），而是帶你理解**背後的原理**，
希望對高中生也友善、看得懂。

## 這個工具在做什麼？

撞球（花式撞球）一共 16 顆：1 號黃色、2 號藍色⋯⋯8 號黑色，9~15 號是跟 1~7 號同色系的
「花色球」（球上有一條色帶，其餘部分是黑色），另外還有一顆母球（白球）。

如果你隨便給我看一張撞球的照片，就算照片上的數字被轉到背面看不到，人類還是可能靠「這是
紅色、而且看起來有一大片黑色」判斷出這是 11 號而不是 3 號——因為 3 號整顆都是紅色，
11 號則是紅色色帶 + 大片黑色。這個網頁工具做的，就是把這種「靠顏色和圖案推理」的過程，
變成一段可以被電腦執行、可以被驗證的演算法。

## 為什麼不能直接用照片，而要「建模」？

一開始最直覺的做法是：把一張撞球的照片剪一小塊圓形下來，貼到 3D 球體上當作材質。這樣做
有一個致命的問題——**一張照片只拍到球的正面，球的背面永遠是空白的**。

想像你把地球的照片剪成一個圓形貼紙，貼在一顆籃球的正面，籃球轉到背面的時候，你只會看到
一片空白，因為那些資訊照片裡根本沒拍到。

所以這個專案改成：**用程式碼「畫」出每一顆球完整的展開圖**，涵蓋球的正面、背面、南極、
北極，缺一不可。這樣不管你把 3D 球轉到哪個角度，球面上永遠都有正確的顏色資訊可以分析。

### 展開圖是怎麼「包」到球上的？

這張展開圖的技術名稱叫**等距柱狀投影（equirectangular projection）**，跟世界地圖的畫法
一樣：把球攤平成一張長方形圖片，橫軸（x）對應「經度」（繞球轉一圈，0°~360°），縱軸（y）
對應「緯度」（從北極到南極）。Three.js 的 `SphereGeometry` 預設就是用這種方式把材質貼到
球面上，所以我們只要用 `<canvas>` 畫一張長方形圖，三維球體就會自動把它正確地包覆起來——
横向的色帶會變成繞著球「腰部」一圈的條紋，跟真正的花色撞球一樣。

程式碼在 [`src/ballTexture.js`](src/ballTexture.js)，用 Canvas 2D API 畫出：

- **實色球（1~8 號）**：整張圖填滿該顏色，正面、背面各畫一個白底黑框的數字圓標。
- **花色球（9~15 號）**：整張圖先填**黑色**（不是傳統美式撞球那種白色，這是照參考照片裡
  這組球實際的塗裝去畫的），中間橫著畫一條該顏色的色帶（也就是球的「腰部」），再疊上白色
  數字圓標。
- **母球**：整張圖填奶白色，加一個小黑點。

## 核心原理：怎麼「不讀數字」判斷幾號球？

這是整個專案最重要的部分，寫在 [`src/colorAnalysis.js`](src/colorAnalysis.js)。

### 第一步：把顏色從 RGB 轉成 HSV

電腦畫面上的顏色通常用 RGB（紅、綠、藍三個 0~255 的數字）表示，但 RGB 不適合拿來判斷
「這是什麼顏色」，因為同一種紅色，在亮的地方跟暗的地方，RGB 數值可能完全不一樣。

HSV 是另一種描述顏色的方式，把顏色拆成三個更符合人類直覺的維度：

| | 意思 | 範圍 |
|---|---|---|
| **H**ue 色相 | 這是什麼顏色（紅、黃、綠、藍⋯⋯），想像成一個色環 | 0°~360° |
| **S**aturation 飽和度 | 顏色有多「純」，0 是灰階，1 是最鮮豔 | 0~1 |
| **V**alue 明度 | 顏色有多亮，0 是黑色，1 是最亮 | 0~1 |

轉成 HSV 之後，**同一顆紅球不管在亮處還是暗處，色相 H 幾乎都維持在同一個數值附近**（大約
355°~5°），這就是為什麼要先做這個轉換——它讓「顏色分類」這件事，不會被場景裡的光影變化
干擾太多。

### 第二步：把畫面上的每個像素分成三類

拿到目前鏡頭看到的那一面（怎麼拿到的，下一節會講），程式會掃過每一個像素，依照它的 HSV
值分成三桶：

```js
if (v < 0.22)              → 記一次「黑色」
else if (s < 0.32 && v > 0.55) → 記一次「白色」
else                        → 記一次「彩色」，並依色相 H 累加進對應的色相統計格
```

掃完之後，程式知道畫面裡「黑色佔多少比例」「白色佔多少比例」「哪一個色相最多」。

### 第三步：用這三個比例做判斷

```
黑色比例 > 50%？             → 8 號黑球
白色比例很高、幾乎沒有彩色？   → 母球（白球）
否則 → 看主要色相是哪一色（黃/藍/紅/粉紅/紫/綠/棕）
      再看黑色比例是不是「明顯偏高」（> 12%）
        是 → 花色球（例如紅色 + 黑色偏高 = 11 號）
        否 → 實色球（例如紅色 + 幾乎沒黑色 = 3 號）
```

這裡的關鍵洞察是：**實色球唯一的黑色來源，是數字圓標上那個細細的黑框跟數字本身**（大概
只佔畫面 2%~5%）；花色球只要鏡頭多少看到一點黑色的球身，黑色比例就會明顯跳升。這個「多一點
黑色 vs. 幾乎沒有黑色」的落差，就是分辨 3 號和 11 號的關鍵——**跟數字印在哪一面完全無關**。

### 為什麼「即時」分析、而不是拍一次照就固定？

程式並不是「選球的那一瞬間拍一張照片分析完就結束」，而是每秒重新分析大約 7~8 次，資料來源
永遠是「目前 3D 攝影機正對著的那一面」。這是用 Three.js 的 `WebGLRenderTarget`（一塊看
不見的畫布，跟畫面上顯示的球是同一個場景、同一台攝影機，只是額外算一次畫面給程式讀取像
素）做出來的效果。

這也是為什麼你拖曳旋轉球體時，下方的判定結果會跟著即時改變——如果你把紅色的 11 號轉到只
剩下一塊純紅色、完全看不到黑色的角度，程式會很老實地告訴你「看起來像 3 號」，因為從那個
角度看，兩顆球給出的顏色資訊真的一樣；只要再轉一點點，讓黑色球身露出來，答案就會立刻變回
11 號。這不是 bug，而是誠實反映「顏色分析在這個角度能拿到多少資訊」。

## 專案結構

```text
.
├── index.html            # 版面、文字說明
├── src/
│   ├── main.js            # 點選球、Three.js 場景、即時分析迴圈
│   ├── ballTexture.js      # 用 Canvas 2D 畫出每一顆球的完整展開圖
│   └── colorAnalysis.js    # HSV 轉換、像素統計、分類判斷（本文重點）
├── public/pool_balls.jpg   # 左側「挑球用」的參考照片（純粹當按鈕，不參與分析）
├── vite.config.js          # 建構設定
├── flake.nix / Justfile    # NixOS 開發環境與常用指令
└── .github/workflows/      # 自動部署到 GitHub Pages
```

## 本機執行

```bash
nix develop      # 進入含 bun 的開發環境
just dev         # 啟動開發伺服器
```

開啟終端機顯示的網址（預設 http://localhost:5173），點選左側照片裡任何一顆球即可。

## 已知的限制（誠實說明）

- **棕色（7/15）跟綠色（6/14）的判斷相對脆弱**：這兩個顏色本身偏暗，燈光造成的反光更容易
  把它們的飽和度「洗」得比較低，跟白色的門檻比較接近。目前用調整燈光強度跟材質粗糙度的方式
  緩解，但如果未來又觀察到誤判，這是第一個該檢查的地方。
- **顏色分類的色相分界，是照這組球實際畫出來的顏色去校準的**，不是查表得到的「教科書標準
  撞球色」——因為實測發現參考照片裡這組球的配色跟教科書不同（4 號是粉紅色、5 號是紫色，
  沒有橘色）。如果之後想套用到別的球具照片，可能需要重新校準色相分界。
